🛡️ AI Security Scan

Powered by AI. Collects your full cluster inventory — workloads, networking, ConfigMaps, Secrets, RBAC and service accounts — and performs a comprehensive security audit aligned with the CIS Kubernetes Benchmark v1.8 and NSA/CISA Kubernetes Hardening Guidance.

What AI Scans:

🔒 Pod Security Privileged containers, host PID/IPC/Network sharing, root UIDs, allowPrivilegeEscalation, readOnlyRootFilesystem, dangerous Linux capabilities, hostPath mounts
🌐 Network Policy Missing NetworkPolicy (unrestricted pod-to-pod traffic), overly permissive policies (0.0.0.0/0 CIDRs)
🔑 RBAC ServiceAccounts with automountToken enabled, RoleBindings granting cluster-admin or wildcard verbs, pods using the default ServiceAccount
📦 Image Security Images using :latest tag, images pulled from Docker Hub without a registry prefix (supply chain risk)
🗝️ Secrets Management Plaintext secrets in environment variables (not secretKeyRef), overloaded Opaque Secrets with too many keys
📋 ConfigMap Security ConfigMaps storing sensitive keys (passwords, tokens, private keys) — these should use Kubernetes Secrets instead
💻 Resource Management Missing CPU/memory limits (DoS risk), missing resource requests (scheduling instability)
⏱️ Availability Missing readiness/liveness probes, single-replica critical workloads (no HA), missing PodDisruptionBudgets
Note: Without AI configured (GOOGLE_CLOUD_PROJECT env var), the scan runs deterministic checks only. With AI, it performs contextual AI analysis and provides ai_insight explanations for each finding.

💰 AI Resource Optimizer

Powered by AI. Analyses all workload specs (Deployments, StatefulSets, DaemonSets, Jobs) and — since no metrics server is available — estimates realistic CPU and memory usage from workload type, image name, and engineering heuristics. Then calculates costs and suggests right-sized resource configs.

🏷️ Cost Model (org-specific)

Rate €60 per CPU core per month
Billing basis billable = max(cpu_requests, actual_cpu_usage)
If requests > usage → billed on requests (over-provisioned). If usage > requests → billed on usage (under-provisioned).
Memory Not billed separately, but OOM kills cause incidents — AI still recommends appropriate limits.

📊 Recommendation Types

📉 Cost Saving CPU requests far exceed estimated usage — over-provisioned, billed on requests. Reducing them saves money.
📈 Performance Risk Estimated CPU usage exceeds requests — under-provisioned, billed on usage and at risk of throttling.
⚠️ Stability Risk Missing resource limits or requests — pod can consume unlimited resources, destabilising the node.
✅ Right-Sized Requests closely match estimated usage with appropriate headroom — no immediate action needed.
Suggested Config formula: cpu_request = estimated_usage × 1.3 (30% headroom) · cpu_limit = cpu_request × 2 (burst)
